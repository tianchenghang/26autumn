# Internationalization

swifty-docs supports internationalization through directory-based locale separation. Each locale lives in its own subdirectory with its own Markdown files, sidebar configuration, and navigation. The scanner treats each locale as an independent docs tree, so translated content is built and routed the same way as the default locale.

## Directory Structure

The standard approach is to place each locale under a language-code directory inside the main docs folder:

```
docs/
├── en/
│   ├── index.md
│   ├── guide/
│   │   ├── getting-started.md
│   │   └── configuration.md
│   └── api/
│       └── overview.md
├── zh/
│   ├── index.md
│   ├── guide/
│   │   ├── getting-started.md
│   │   └── configuration.md
│   └── api/
│       └── overview.md
└── ja/
    ├── index.md
    └── guide/
        └── getting-started.md
```

Each locale directory is a self-contained docs tree. Files in `docs/en/` produce routes under `/docs/en/`, files in `docs/zh/` produce routes under `/docs/zh/`, and so on. The scanner walks the entire docs directory recursively, so locale directories are discovered automatically without any registration step.

## Configuration

Because swifty-docs uses a single `swifty-docs.config.ts` file, all locales share the same configuration object. The sidebar and nav are configured per path prefix, which maps naturally to locale directories.

### Sidebar per locale

Use `"auto"` for each locale prefix to generate sidebars independently from each locale's directory structure:

```ts
import { defineConfig } from "swifty-docs/vite";

export default defineConfig({
  docs: "docs",
  baseUrl: "/docs/",
  title: "My Library",

  sidebar: {
    "/docs/en/": "auto",
    "/docs/zh/": "auto",
    "/docs/ja/": "auto",
  },
});
```

Each locale's sidebar is built from its own directory tree. Files in `docs/en/guide/` appear only in the `/docs/en/` sidebar, and files in `docs/zh/guide/` appear only in the `/docs/zh/` sidebar. The `sidebar_position` frontmatter field works independently within each locale.

### Navigation with locale switching

The top navbar can include a locale switcher as a dropdown item. Each nav item points to the corresponding locale's entry page:

```ts
export default defineConfig({
  nav: [
    { text: "Guide", link: "/docs/en/guide/getting-started" },
    { text: "API", link: "/docs/en/api/overview" },
    {
      text: "Language",
      link: "#",
      items: [
        { text: "English", link: "/docs/en/" },
        { text: "中文", link: "/docs/zh/" },
        { text: "日本語", link: "/docs/ja/" },
      ],
    },
  ],
});
```

External links open in a new tab automatically. Internal locale links use client-side navigation through the swifty-mvc router.

## Slugify and Unicode Support

Heading anchors and route slugs are generated by the `slugify()` function, which uses Unicode property escapes to preserve non-ASCII characters. This means headings written in Chinese, Japanese, Korean, Cyrillic, Arabic, or any other script produce valid, readable anchor slugs.

### How it works

The slugify implementation applies the following rules in order:

1. Lowercase the text.
2. Replace any character that is not a Unicode letter (`\p{L}`), Unicode number (`\p{N}`), whitespace, or hyphen with a dash.
3. Replace whitespace sequences with a single dash.
4. Collapse consecutive dashes into one.
5. Trim leading and trailing dashes.
6. Prefix a leading digit with an underscore so the slug is a valid CSS selector.

The key detail is step 2: the regex `/[^\p{L}\p{N}\s-]/gu` uses the `u` (Unicode) flag with property escapes. This preserves letters and numbers from all scripts while stripping punctuation and symbols.

### Examples

| Input                  | Output               |
| ---------------------- | -------------------- |
| `"Getting Started"`    | `"getting-started"`  |
| `"配置指南"`           | `"配置指南"`         |
| `"はじめに"`           | `"はじめに"`         |
| `"Настройка"`          | `"настройка"`        |
| `"API Reference (v2)"` | `"api-reference-v2"` |
| `"123 Setup"`          | `"_123-setup"`       |
| `"Hello! World"`       | `"hello-world"`      |

CJK characters pass through unchanged, which means anchor links like `#配置指南` work correctly in browsers. The leading-digit prefix ensures that slugs like `_123-setup` are valid CSS identifiers for `querySelector()`.

### Anchor links in translated content

Because slugify preserves Unicode characters, heading anchors in translated pages use the translated text. A Chinese page with `## 快速开始` produces the anchor `#快速开始`, and an English page with `## Quick Start` produces `#quick-start`. This is the expected behavior: anchor links point to headings in the same language as the page content.

If you need to link from one locale to a specific heading in another locale, use the full path including the slug:

```markdown
See the [Chinese configuration guide](/docs/zh/guide/configuration#基本选项).
```

## Best Practices

### Keep locale directories parallel

Mirror the directory structure across locales. If `docs/en/guide/configuration.md` exists, create `docs/zh/guide/configuration.md` and `docs/ja/guide/configuration.md` at the same relative path. This makes it easy to find corresponding translations and keeps the sidebar structure consistent.

### Use frontmatter for translated metadata

Each locale's files carry their own frontmatter. Translate the `title`, `description`, and `sidebar_label` fields directly:

```yaml
---
title: 配置
description: swifty-docs 配置参考
sidebar_label: 配置
sidebar_position: 2
---
# 配置

本页介绍所有配置选项。
```

The `sidebar_position` field can differ across locales if you want to reorder pages for a specific audience, though keeping the same order simplifies maintenance.

### Handle incomplete translations gracefully

Not every page may be translated immediately. The scanner only generates routes for files that exist, so a missing translation simply does not appear in that locale's routes or sidebar. There is no error or fallback behavior to configure.

To guide users toward available content, consider adding a note at the top of partially-translated sections or linking back to the default locale for untranslated pages:

```markdown
::: tip
This page is not yet available in Japanese. See the [English version](/docs/en/api/router).
:::
```

### Avoid hardcoding locale paths in content

When linking between pages within the same locale, use relative paths that do not include the locale prefix. This makes content portable if you restructure locale directories:

```markdown
<!-- Good: relative within the same locale -->

See [Configuration](./configuration) for details.

<!-- Avoid: hardcoded locale prefix for same-locale links -->

See [Configuration](/docs/en/guide/configuration) for details.
```

Cross-locale links (linking from one language to another) must use absolute paths since they point to a different route prefix.

### Separate site-wide config from translated content

The `title` and `description` in `defineConfig()` apply globally. If you need locale-specific site metadata (such as a translated site title in the navbar), consider one of two approaches:

- Use a single configuration and accept that the navbar title is shared across locales. Translate nav item text directly in the config.
- Run separate `defineConfig()` calls per locale with different `baseUrl` values, each pointing to its own locale directory. This requires multiple build passes but gives full control over per-locale metadata.

### Search across locales

The local search index includes all pages from all locales by default. Users searching in one locale see results from other locales as well, with each result linking to its respective locale path. This is useful for multilingual audiences but may produce mixed-language search results.

If you prefer locale-scoped search, configure separate search instances per locale by building each locale as an independent docs site with its own `defineConfig()` and `baseUrl`.
